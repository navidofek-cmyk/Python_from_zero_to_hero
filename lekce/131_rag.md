# Lekce 131: RAG — Retrieval-Augmented Generation

## 🎯 Co je RAG?

RAG je architektura, kde LLM **nedostane celou znalostní bázi do kontextu**, ale místo toho se nejdřív vyhledají relevantní dokumenty a teprve ty se předají modelu. Výsledek: aktuální, přesné odpovědi i na otázky mimo tréninkový cutoff.

```
Dotaz uživatele
     │
     ▼
┌─────────────┐     ┌──────────────────┐
│  Retriever  │────▶│  Vector store    │
│  (hledání)  │◀────│  (embeddingy)    │
└─────────────┘     └──────────────────┘
     │
     ▼ (relevantní dokumenty)
┌─────────────┐
│     LLM     │──▶ Odpověď
│  (generace) │
└─────────────┘
```

## 🧩 Základní komponenty RAG

### 1. Chunking — rozdělení dokumentů

```python
def rozdel_na_chunky(text: str, velikost: int = 500, prekryv: int = 50) -> list[str]:
    chunky = []
    start = 0
    while start < len(text):
        konec = start + velikost
        chunky.append(text[start:konec])
        start += velikost - prekryv   # překryv zachovává kontext
    return chunky
```

Strategie chunkingu:
| Strategie | Kdy použít |
|-----------|-----------|
| Fixed-size | Rychlé prototypy, homogenní text |
| Sentence-based | Přirozené hranice, Q&A |
| Recursive (LangChain styl) | Strukturované dokumenty |
| Semantic | Nejvyšší kvalita, pomalejší |

**Kosinová podobnost** — jak blízko jsou dva vektory:

```python
import math

def kosinus_podobnost(a: list[float], b: list[float]) -> float:
    dot = sum(x * y for x, y in zip(a, b))
    norma_a = math.sqrt(sum(x ** 2 for x in a))
    norma_b = math.sqrt(sum(x ** 2 for x in b))
    return dot / (norma_a * norma_b)
```

### 3. Vector store — uložení a vyhledávání

Jednoduchý in-memory vector store:

```python
from dataclasses import dataclass, field

@dataclass
class Dokument:
    id: str
    text: str
    embedding: list[float]
    metadata: dict = field(default_factory=dict)

class VectorStore:
    def __init__(self):
        self._dokumenty: list[Dokument] = []

    def pridej(self, doc: Dokument) -> None:
        self._dokumenty.append(doc)

    def hledej(self, query_embedding: list[float], k: int = 3) -> list[Dokument]:
        scored = [
            (kosinus_podobnost(query_embedding, doc.embedding), doc)
            for doc in self._dokumenty
        ]
        scored.sort(key=lambda x: x[0], reverse=True)
        return [doc for _, doc in scored[:k]]
```

Produkční alternativy:
| Nástroj | Typ | Kdy |
|---------|-----|-----|
| ChromaDB | lokální | vývoj, malé projekty |
| Qdrant | server/cloud | produkce, self-hosted |
| Pinecone | cloud | managed, velké objemy |
| pgvector | PostgreSQL ext. | máš už Postgres |
| FAISS | in-process | vysoký výkon, research |

### 4. Retrieval — vyhledání relevantních chunků

```python
def retrievuj(dotaz: str, store: VectorStore, k: int = 3) -> list[Dokument]:
    query_emb = embed(dotaz)
    return store.hledej(query_emb, k=k)
```

**Reranking** — druhý průchod pro přesnější výsledky:

```python
def rerank(dotaz: str, dokumenty: list[Dokument]) -> list[Dokument]:
    # Cross-encoder score (nebo LLM relevance judge)
    scored = [(relevance_score(dotaz, d.text), d) for d in dokumenty]
    return [d for _, d in sorted(scored, reverse=True)]
```

### 5. Augmented Generation — předání kontextu LLM

```python
import anthropic

def rag_odpoved(dotaz: str, store: VectorStore) -> str:
    dokumenty = retrievuj(dotaz, store, k=3)

    kontext = "\n\n---\n\n".join(
        f"[Zdroj {i+1}]\n{doc.text}"
        for i, doc in enumerate(dokumenty)
    )

    prompt = f"""Na základě následujících dokumentů odpověz na otázku.
Pokud odpověď není v dokumentech, řekni to.

Dokumenty:
{kontext}

Otázka: {dotaz}"""

    client = anthropic.Anthropic()
    zprava = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        messages=[{"role": "user", "content": prompt}]
    )
    return zprava.content[0].text
```

## 🔧 RAG pipeline — kompletní přehled

```
1. INDEXACE (jednou / při změně dat)
   Dokumenty → Chunking → Embedding → Vector Store

2. DOTAZOVÁNÍ (při každém dotazu)
   Dotaz → Embedding → Similarity Search → Top-K chunků
         → (Reranking) → Prompt sestavení → LLM → Odpověď
```

## ⚠️ Časté problémy RAG

| Problém | Příčina | Řešení |
|---------|---------|--------|
| Halucinations i s RAG | LLM ignoruje kontext | Přísnější system prompt, self-check |
| Špatný retrieval | Chunky příliš velké/malé | Experimentuj s velikostí, overlap |
| Pomalá indexace | Mnoho dokumentů | Dávkové embeddingy, async |
| Zastaralé embeddingy | Data se mění | Inkrementální re-indexace |
| Lost in the middle | LLM ignoruje střed kontextu | Reranking, méně chunků |

---

## 🚀 Pokročilé techniky

- **HyDE** (Hypothetical Document Embeddings) — LLM nejdřív vygeneruje hypotetickou odpověď, tu embeduj a hledej podobné dokumenty
- **Multi-query retrieval** — jeden dotaz → více variant dotazů → více výsledků → deduplikace
- **Parent-child chunking** — hledej v malých chuncích, vrať velký rodičovský chunk
- **Self-RAG** — LLM sám rozhodne, kdy potřebuje retrieval

---

## 📝 Shrnutí

- RAG = hledej relevantní dokumenty PŘED generací
- Klíčové komponenty: chunking, embedding, vector store, retrieval, generace
- Kvalita závisí nejvíc na: kvalitě chunkingu + kvalitě embeddings
- Jednoduchý RAG funguje dobře — přidávej složitost jen když potřebuješ
